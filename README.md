# wisewater-website
Design Thinking in IT (2008ICT) - Assessment 2 - Water Usage Website

---

## **Project Overview**

This repository contains the front-end prototype for the WiseWater website, developed as part of the Design Thinking assignment.

The goal of this project is to create a simple, intuitive, and user-friendly website that helps renters in South East Queensland understand and reduce their water usage.

This is a front-end only project using:

- HTML  
- CSS  
- Minimal JavaScript (only if necessary, subject to clarification)

---

## **Project Structure**

- `index.html`

## **Team Workflow**

### Development Approach
Each team member is responsible for one specific part/page of the website.

### **Assignment Strategy**
- Work individually on your assigned page  
- Focus on structure first, then styling  
- Upload your work when your section is complete  
- At the end, one person will merge and clean everything  

---

## **Page Responsibilities (Example Split)**

- `index.html` → Landing page  
- `login.html` → Login form  
- `dashboard.html` → Main logged-in view  
- `calculator.html` → Water usage inputs  
- `resources.html` → Educational guides  
- `alerts.html` → Alerts & support  

**Additional responsibilities:**
- One person handles global CSS (`styles.css`)  
- One person performs final consistency pass  

---

## **Rules**

- Do NOT overwrite other people’s files  
- Only edit files you are responsible for  
- If unsure → ask before changing shared files  

---

## **Code Standards**

### HTML Standards
- Use proper structure (e.g. `<header>`, `<section>`, `<footer>`)  
- Keep code clean and readable  
- Use indentation consistently  
- Avoid unnecessary complexity  

### CSS Standards
- Keep styling in `styles.css`  
- Use simple class names:
  - `.nav`
  - `.hero`
  - `.btn`
- Avoid inline styles  
- Maintain consistent spacing and colours  

---

## **Design Standards (Critical for Marks)**

The website must be:

- Simple  
- Clear  
- Easy to navigate  
- Not overwhelming  

### Avoid:
- Large blocks of text  
- Over-complicated layouts  
- Too many features  

### Focus on:
- Readability  
- Visual hierarchy  
- Clean layout  

---

## **JavaScript Usage**

JavaScript should be:

- Minimal  
- Only used if required (e.g. calculator)  
- If unsure → do not use JavaScript  

---

## **Final Integration Process**

At the end of development:

### **One team member will:**
- Combine all pages  
- Fix broken links  
- Ensure consistent styling  
- Clean up code  

### **Team will:**
- Review together  
- Test navigation  
- Check visual consistency  

---

## **What NOT to Do**

- Do NOT build a backend  
- Do NOT overcomplicate features  
- Do NOT use frameworks (React, etc.)  
- Do NOT change other people’s work without discussion  

---

## **Definition of _Done_**

A page is complete when:

- It is structured correctly  
- It follows design guidelines  
- It is styled consistently  
- It links properly to other pages  

---

## **Final Goal**

A clean, simple, fully navigable prototype website that demonstrates:

- Good UI design  
- Clear user flow  
- Strong alignment with the assignment requirements  

---

## **Key Principle**

**Keep it simple. If it feels complex, it’s probably wrong.**
